## 2. Перечень выполненных задач

### 2.1 Backend-функциональность

#### Feature: Category Model (#7)

* Создана SQLAlchemy-модель Category
* Добавлены все необходимые поля
* Настроены связи с User и Task 
* Добавлены индексы 
* Создана и применена Alembic-миграция
* Обеспечена уникальность категории у одного пользователя

**Результат:** Модель категорий создана, исправлена после ревью и готова к использованию в API.

---

#### Feature: Create Category (#14)

* Реализован эндпоинт POST /api/categories
* Добавлена Pydantic-схема CategoryCreate
* Категория создается и привязывается к текущему пользователю
* Добавлена проверка существования категории у пользователя

**Результат:** Категория создается корректно с учетом авторизации, проверка уникальности имени категори у пользователя работает.

---

#### Feature: Update Category (#20)

* Реализован эндпоинт PUT /api/categories/{category_id} 
* Создана Pydantic схема CategoryUpdate
* Настроена проверка существования категории
* Настроена проверка наличия прав доступа к категории
* Настроена проверка уникальности нового имени категории (при изменении)

**Результат:** Релизовано обновление категории по ID, если пользователь авторизован, имеет права доступа к категории и, если имя изменяется, то оно уникально у этого пользователя среди других категорий

---

#### Feature: Get Single Task (#23)

* Реализован эндпоинт GET /api/tasks/{task_id}
* Возвращает задачу, только если она принадлежит текущему пользователю
* Возвращает полные данные задачи
* Включает данные связанной категории (если такая существует есть)

**Результат:** Получение одной задачи пользователя по ID, если она существует у текущего пользователя

---

#### Feature: Delete Task (#26)

* Реализован эндпоинт DELETE /api/tasks/{task_id}
* Проверяется авторизация пользователя
* Проверяется существование задачи по ID
* Проверяется право доступа к задаче у пользователя
* Задача удаляется из БД

**Результат:** Задача успешно удаляется из БД при выполнении всех условий: пользователь авторизован, задача существует, пользователь имеет права доступа к задаче.

---

#### Feature: Search Tasks by Title (#46)

* Добавлен query параметр search в GET /api/tasks эндпоинт.
* Реализован поиск по полю title (case-insensitive).
* Комбинируется с существующими фильтрами и пагинацией
* Пустой search не влияет на результат

**Результат:** Поиск по полю title работает корректно 

---


### 2.2 Тестирование


#### QA: Test Data Preparation (#31)

* Создан Python скрипт для генерации тестовых данных
* Можно запустить командой python scripts/seed_data.py
* Генерация тестовых пользователей
* Генерация категорий для каждого пользователя
* Генерация задач для каждого пользователя
* Скрипт идемпотентный (при повторном запуске дублей не создается)

**Результат:** Скрипт создает реалистичные, разнообразные и полезные данные в БД.

---

#### Manual Testing: Task Flow (#36)

* Создана Postman коллекцию для всех эндпоинтов
* Настроены переменные среды (base_url, token)
* Созданы requests для всех эндпоинтов
* Протестирован полный сценарий: регистрация → login → создание задачи → чтение → обновление → удаление 
* Протестирована валидация (некорректные данные)
* Протестирована авторизация (запросы без токена)
* Протестированы права доступа (один пользователь не может видеть задачи другого)
* Отчет по ручному тестированию сохранен в docs/manual-testing-report.md
* Postman коллекция сохранена в docs/postman/
* Инструкция по импорту коллекции Postman сохранена в docs/postman/

**Результат:** Создана коллекция в Postman с запросами для всех эндпоинтов, а также протестирован полный сценарий пользовательских действий, валидация, авторизация,права доступа
